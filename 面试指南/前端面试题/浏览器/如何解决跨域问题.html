    如何解决跨域问题

    zsy 2021/12/12

    # 前话

    首先理清楚啥是同域：如果地址的 协议，域名，端口 都是一样的，那么就称是同域

    补充：下面两个url地址的对比

    • www.af.cn:8080 一级域名
    • a.af.cn:8080 二级域名

    那么为什么浏览器不支持跨域呢？

    • 没有同源策略：即支持跨域
    • 首先 cookie 和 LocalStorage 是不支持跨域的
    • DOM也有同源策略，比如那个 iframe
    • ajax 也不支持跨域

    下面是实现跨域的九种手段

    • jsonp
    • cors
    • postMessage : 一般是使用 iframe ，一个页面嵌套另外一个页面，这两个页面之间进行通信
    • window.name
    • location.hash
    • http-proxy ：webpack中的跨域代理的本质就是 http-proxy
    • nginx
    • websocket
    • doucment.domain : 专门实现 子域 与 父域

    补充说一下哈：在浏览器中的能看到XHR，说明就是和 ajax 有关

    # jsonp

    • 原理：因为script标签发请求不受同源策略的限制，所以可以通过动态构建script节点，利用节点的src属性，发出get请求，从而绕开ajax引擎
    • 具体实现：前端通过在请求的url指定一个回调函数，同时在window上声明这个函数，然后服务器返回该函数调用，同时传入参数，因为是使用script标签请求的，所以会把后端返回的数据当作js脚本执行，那么就调用了原先在window上已经定义了的这个函数
    • 有种前端定义函数，服务器端‘调用函数’的感觉
    • 缺陷：该方式只能用于get请求

    在1.html:

    <!DOCTYPE html>
<html lang="en">

<head>
    <meta charset="UTF-8">
    <meta http-equiv="X-UA-Compatible" content="IE=edge">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>

<body>
    <script>
        // 其实本质就是利用 script 的 src属性，然后把 script 脚本执行，然后再干掉 添加的 script 标签
        // json 只能发送get请求，不支持post put delete，并且不安全，可能有 xss 攻击
        function jsonp({
            url,
            params,
            cb
        }) {
            return new Promise((resolve, reject) => {
                let script = document.createElement('script')
                params = {
                    ...params,
                    cb
                }
                let arrs = []
                for (const key in params) {
                    arrs.push(`${key}=${params[key]}`)
                }
                script.src = `${url}?${arrs.join('&')}`
                document.body.appendChild(script)
                window[cb] = function (data) {
                    resolve(data)
                    document.body.removeChild(script)
                }
            })
        }
        jsonp({
            url: 'http://localhost:3000/say',
            params: {
                mes: '我爱你',
            },
            // 这个 cb 其实就是个函数 show() ,会挂载到window上执行
            cb: 'show'
        }).then(
            data => {
                console.log(data);
            }
        )
    </script>
</body>

</html>

    # CORS

    • 全称 : “跨域资源共享” （Cross-Origin Resource Sharing）
    • 使用cors的方式,需要在服务器端配置
    • 浏览器将cors请求分成两类：
      • 简单请求：浏览器发出请求，就会在该请求的头信息中，增加一个origin字段，这个origin字段说明了本次请求来自哪个源，服务器端设置Access-Control-Allow-Orgin，指定允许该源允许跨域，这样ajax引擎就不会拦截本次响应，这样就解决了跨域问题了
      • 非简单请求:比如发送put请求，那么浏览器会发出一次预检请求，判断该域名是否在服务器的白名单中，如果收到肯定回复后才会发起请求

    在index.html:

    <!DOCTYPE html>
<html lang="en">

<head>
    <meta charset="UTF-8">
    <meta http-equiv="X-UA-Compatible" content="IE=edge">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>

<body>
    <script>
        let xhr = new XMLHttpRequest
        // true 表示支持异步
        xhr.open('PUT', 'http://localhost:4000/getlist', true)
        // 设置请求头
        xhr.setRequestHeader('name', 'zsy')
        // cookie默认是不允许跨域的，那么服务器就不会收到 cookie
        document.cookie = 'name=zsycookie'
        // 强制凭证 ： 强制将 cookie 携带过去，同时服务器那边也要设置，才能接收到 cookie
        xhr.withCredentials = true
        xhr.onreadystatechange = function () {
            // 4 表示获取服务器数据完成
            if (xhr.readyState === 4) {
                // 304 表示走缓存
                if (xhr.status >= 200 && xhr.status < 300 || xhr.status === 304) {
                    // 默认是会报错的 ：index.html:27 Refused to get unsafe header "name"
                    // 因为浏览器会认为服务器返回的请求头中的 name 是不安全的
                    console.log(xhr.getResponseHeader('name'))
                    console.log(xhr.response)
                }
            }
        }
        xhr.send()
    </script>
</body>

</html>

    在 server2.js:

    const express = require('express')
const app = express()

// 白名单
const whiteList = ['http://localhost:3000']

app.use(function (req, res, next) {
    let origin = req.headers.origin
    // 百名单里面包含有 origin 这个请求源
    if (whiteList.includes(origin)) {
        // 设置哪个源可以访问我
        res.setHeader('Access-Control-Allow-Origin', origin)
        // 支持前端给请求头添加 name 属性
        res.setHeader('Access-Control-Allow-Headers', 'name')
        // 设置后，服务器端就可以拿到前端携带的 cookie
        res.setHeader('Access-Control-Allow-Credentials', true)
        // 设置之后，服务端返给浏览器的请求头，前端就可以展示到页面了
        res.setHeader('Access-Control-Expose-Headers', 'name')
        // 默认只支持 get post head 请求，设置后就支持 put 请求了，但是注意 put 请求之前，会有一个
        // 请求方法为 OPTIONS 的请求(预检测)，那么就会导致前端发送 put 请求时，会走两次请求
        res.setHeader('Access-Control-Allow-Methods', 'PUT')
        // 发送一次请求后，三秒内不会进行预检测,叫做 【预检的存活时间】
        res.setHeader('Access-Control-Max-Age', 3000)

        // 注意如下：如果第一个设置了 * 那么就不能有 Credentials
        res.setHeader('Access-Control-Allow-Origin', '*')
        res.setHeader('Access-Control-Allow-Credentials', '*')
    }
    next()
})


app.put('/getlist', function (req, res) {
    res.setHeader('name', 'peiqitou')
    res.send('我不爱你哈哈哈')
})


app.get('/getlist', function (req, res) {
    // 可以看到请求的源 ： origin: 'http://localhost:3000'
    // console.log(req.headers)
    res.send('我不爱你哈哈哈')
})


app.listen(4000, function () {
    console.log('server2启动成功')
})

    # window.name

    • 基于同一个窗口中设置了window.name 后不同源的页面也可以访问
    • 假如有：a b 页面同源、 c 不同源
    • 实现原理：在父级页面（a页面）使用iframe引用c页面，在不同源的子页面（c页面）设置window.name=xxx,然后将父级页面引用的c页面修改为一个和父级同源的页面（b页面），这个时候父级页面（a页面）就可以访问同源的子页面中window.name的数据了
    • 该方式的好处：可以传输的数据量大

    在 a.html:

    <!DOCTYPE html>
<html lang="en">

<head>
    <meta charset="UTF-8">
    <meta http-equiv="X-UA-Compatible" content="IE=edge">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>

<body>
    <!-- 
        说明：
        a 和 b 是在同域下的
        c 不与 a和b 同域

        实现的思路：
        1.把 c 嵌套到 a 中，当 c 加载，同时在window上挂 name 属性
        2.把 b 嵌套进来，获取 window.name

     -->
    <iframe src="http://localhost:4000/c.html" frameborder="0" onload="load()" id="demo"></iframe>
    <script>
        let first = true
        let iframe = document.getElementById('demo')

        function load() {
            if (first) {
                iframe.src = 'http://localhost:3000/b.html'
                first = false
            } else {
                console.log(iframe.contentWindow.name)
            }
        }
    </script>
</body>

</html>

    # location.hash

    • 在主页面中，通过iframe给不同源的窗口传递hash值，这样单向通信就实现了
    • 那么如果实现双向通信呢？
      • 由于iframe是没有办法访问到不同源的父级窗口的，所以不能直接修改父级窗口的hash值来实现双向通信，这时需要iframe中再加入一个iframe，该iframe和主页面是同源的，然后通过window.parent.parent.location.hash 来修改主页面的hash值。

    在 a.html :

    <!DOCTYPE html>

<head>
    <meta charset="UTF-8">
    <meta http-equiv="X-UA-Compatible" content="IE=edge">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>

<body>
    <!-- 
        实现思路：
        1. 在 a 中将哈希 传给 c
        2. c 传给 b
        3. a 接收 b的

     -->

    <iframe src="http://localhost:4000/c.html#iloveyou" frameborder="0"></iframe>
    <script>
        console.log(location.hash)
    </script>
</body>

</html>

    # websocket

    • 使用websocket协议，因为这个协议没有同源限制

    安装：npm i ws

    在 a.html :

    <!DOCTYPE html>
<html lang="en">

<head>
    <meta charset="UTF-8">
    <meta http-equiv="X-UA-Compatible" content="IE=edge">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>

<body>
    <script>
        // 是高级 api
        let socket = new WebSocket('ws://localhost:3000')
        socket.onopen = function () {
            socket.send('我爱你')
        }
        socket.onmessage = function (e) {
            console.log(e.data)
        }
    </script>

</body>

</html>
